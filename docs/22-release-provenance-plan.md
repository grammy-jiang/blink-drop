# Blink-Drop — Release Provenance + SBOM Plan

| | |
|---|---|
| **Status** | Draft v0.1 — sign + document the released offline sender (2026-07-10) |
| **Scope** | Attach a **build-provenance attestation** and a **CycloneDX SBOM** to the offline single-file sender that `release.yml` already ships. **No product/wire change** — release automation only. |

## 1. Problem

Everything *upstream* of the release is hardened — pinned action SHAs, CodeQL,
gitleaks, OpenSSF Scorecard, `dependency-review`, `harden-runner`,
`persist-credentials: false`. But the artifact people actually download —
`blink-drop-sender-<tag>.html`, uploaded raw by `release.yml` — carries **no
verifiable link back to the build and no bill of materials**. A downloader
cannot answer "what commit / CI run produced this?" or "what libraries are
inlined in it?".

## 2. Approach — two GitHub-native steps, no external service

Add to the existing `attach-offline-sender` job in `.github/workflows/release.yml`:

1. **Build-provenance attestation** — `actions/attest-build-provenance` (v4.1.1,
   SHA `0f67c3f4856b2e3261c31976d6725780e5e4c373`). Produces a Sigstore-signed
   SLSA provenance statement binding the HTML's digest → the exact repo, commit,
   and workflow run. Verifiable offline-of-our-infra with
   `gh attestation verify blink-drop-sender-<tag>.html --repo grammy-jiang/blink-drop`.
2. **SBOM + SBOM attestation** — generate a **CycloneDX** SBOM of the shipped
   bundle with `@cyclonedx/cyclonedx-npm` (6.0.0, dev dependency, pinned in the
   lockfile), then attest it to the artifact with `actions/attest-sbom` (v4.1.0,
   SHA `c604332985a26aa8cf1bdc465b92731239ec6b9e`). Both the SBOM JSON and the
   attestation are attached to the release.

### 2.1 SBOM scope — `--omit dev`

The shipped artifact is one self-contained HTML file; only the **7 runtime
`dependencies`** (`@ngraveio/bc-ur`, `fflate`, `hash-wasm`, `jsqr`,
`qrcode-generator`, `buffer`, `process`) + their transitive tree are inlined by
Vite. `cyclonedx-npm --omit dev` therefore yields a BOM of exactly what is *in
the bundle* — not the heavy dev toolchain (vite/stryker/playwright/sharp), which
never ships. Command lives behind an `sbom` npm script so it is reproducible
locally.

## 3. Decisions

| Decision | Choice | Alternatives / why-not |
|---|---|---|
| SBOM generator | `@cyclonedx/cyclonedx-npm` (dev dep) | **syft / `anchore/sbom-action`** — zero npm deps, but downloads a binary at run time and is less npm-accurate. cyclonedx-npm is pinned in *our* lockfile, runs offline, is governed by our existing `dependency-review`/Dependabot/Scorecard, and reads the exact tree that built the artifact. Cost: ~1 dev dep + its tree (Apache-2.0, dev-only, never shipped). Reversible. |
| SBOM format | CycloneDX JSON | SPDX also fine; CycloneDX is the npm-native default and `attest-sbom` accepts it. |
| Attestation subjects | provenance → HTML; SBOM → HTML | Do not attest the SBOM file's own provenance (metadata, not a build output). |
| Step order | **upload assets, then attest** | Attestation binds by digest, independent of storage. Uploading first means a transient Sigstore/Fulcio outage cannot strip the download; a failed attest still turns the job red (visible, re-runnable). Fail-open on *availability*, never on *integrity*. |
| Tag in action inputs | glob `subject-path: web/blink-drop-sender-*.html` | Avoids interpolating the release tag into an action input; fresh runner ⇒ exactly one match. |

## 4. Permissions

The job's least-privilege token gains two scopes (added at the **job** level, not
the workflow root, which stays `contents: read`):

```yaml
permissions:
  contents: write       # already present — upload the release asset
  id-token: write       # NEW — Sigstore OIDC signing
  attestations: write   # NEW — write the attestations
```

`harden-runner` stays `egress-policy: audit` — attestation talks to Fulcio/Rekor
+ the GitHub attestations API; audit records, does not block.

> **Note — action `uses:` steps ignore `defaults.run.working-directory`.** The
> job's default working dir is `web/`, but that only affects `run:` steps.
> `subject-path` / `sbom-path` on the attest actions must be **repo-root
> relative** (`web/…`).

## 5. Files changed

- `web/package.json` — add `@cyclonedx/cyclonedx-npm` dev dep + `sbom` script:
  `cyclonedx-npm --omit dev --output-format JSON --output-file sbom.cdx.json`.
- `web/package-lock.json` — regenerated by `npm install`.
- `.github/workflows/release.yml` — job permissions + generate-SBOM step + two
  attest steps; upload the HTML **and** the SBOM.

## 6. Robustness / verification

- **Local (pre-merge):** `npm run sbom` produces a valid CycloneDX JSON
  (`bomFormat: "CycloneDX"`, non-empty `components`); `actionlint` passes on the
  edited workflow; `dependency-review` on the PR confirms the new dev dep is
  clean + permissively licensed.
- **Cannot dry-run locally:** the attest steps only fire on `release: published`
  (OIDC needs the real GitHub run context). The first real release validates the
  end-to-end flow; verify with `gh attestation verify` + `gh attestation
  list`. Documented here so the first post-merge release is checked, not assumed.
- **No product surface touched** — the sender/receiver bundles, CSP, and the wire
  format are unchanged; SBOM generation reads the lockfile only.

## 7. Out of scope

- Attesting the PWA receiver deployed to Pages (different trust model — a hosted
  site, not a downloadable artifact). Revisit only if we publish the receiver as
  a downloadable bundle.
- Reproducible-build verification (byte-identical rebuild). Provenance says *who
  built it from what*; reproducibility is a separate, larger effort — not
  required for the download-integrity goal here.
